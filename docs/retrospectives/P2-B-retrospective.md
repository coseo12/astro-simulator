- 헤드리스 측정. 실 브라우저 GPU 가속 성능은 P2-D에서 재측정

## 잘 된 것

1. **per-body 스케일 설계** — `cameraDist × 0.008 / realRadius`로 Earth-Moon 분리를 해석적으로 해결. P1 회고 주요 버그 해소.
2. **ThinInstances 도입** — 소행성대 1000개를 단일 draw call로. Babylon 네이티브 API 활용.
3. **JPL 검증 테스트 확장** — 기존 Standish 패턴을 왜소행성·혜성에 그대로 적용. 데이터 추가만으로 자동 검증.
4. **verify-p2b 통합 스크립트** — 5 브라우저 테스트 + bench 회귀를 1명령으로 실행.

## 어려웠던 것

1. **bench 회귀 기준 재설정** — P2-0 ±2fps 기준은 바디 8개 추가 후 유지 불가. 25%로 완화하고 baseline을 P2-B 종료 시점으로 갱신.
2. **헤드리스 측정의 한계** — 소행성대 N=1000이 8fps로 측정되지만 이는 CPU/JS 한계. 실 브라우저 GPU 가속에서는 크게 다를 것. P2-D 재측정 필수.
3. **혜성 역행 궤도 데이터** — Halley(i=162°), Swift-Tuttle(i=113°)는 역행 궤도. 좌표계 컨벤션 확인 후 문제 없었지만 혼동 위험.

## 프로세스 교훈

- **브라우저 3단계 검증 스크립트 이름 규칙** — `browser-verify-<feature>.mjs`로 통일. verify-p2b.mjs가 일괄 실행.
- **baseline 관리 명시적으로** — 마일스톤 완료 시마다 `bench:scene:set-baseline` 호출이 의무 스텝. `docs/benchmarks/asteroid-belt-n-sweep.md`처럼 스윕 문서도 유용.
- **`kind` 확장은 로더 시험부터** — dwarf-planet/comet 추가 시 loader 카운트 테스트가 즉시 감지해 다른 계산 오류 전파 방지.

## P2-C 인수인계

### 즉시 가능

- 질량·속도 파라미터 슬라이더 (store/URL 이미 있음)
- 프리셋 "만약에" 시나리오 (목성 2배 질량 등)
- 시나리오 URL 북마크 (시간 포함)

### 기술 부채

- 소행성대 N=1000은 CPU 바운드 — WASM Kepler batch 또는 GPU compute(P3)
- 혜성 비중력 효과(태양풍) 미반영 — Halley ±2% 정확도 한계
- per-body 스케일 전환이 시각적으로 갑작스러울 수 있음 — smooth easing 고려

### 설계 검토 필요

- 파라미터 변경 시점부터 시간 역행 불가 (#88 회고에 기록). "리셋" UI 경로 필수.
- N-body 확장 시 소행성대 합류 여부 — 현재는 Kepler 전용. 동시 표시 시 섭동 차이 명확.

## 결론

P2-B는 태양계 콘텐츠를 10 → 18 바디로 확장하고 P1 회고 주요 버그(Moon 가시성)를 해소했다. `?belt=N` 파라미터로 소행성대 규모 실험이 가능해졌고, 거리-의존 스케일로 근접·전체 뷰 모두 사용 가능한 시각적 일관성을 확보했다.

성능 감소(20%)는 콘텐츠 증가 반영분으로 수용하되, P2-D에서 실 브라우저 GPU 성능 재측정 + N-body 대규모 최적화 방향을 재검토한다.
