## 잘 된 것

1. **질량 배수 설계의 단순함** — scene이 map `<id, mul>`만 받고 buildNewton에서 적용. UI store는 동일한 map을 관리. 차분 diff로 전파.
2. **북마크 URL 컨벤션 재사용** — url-sync가 이미 kepler/observe/1일/focus없음을 생략하는 패턴을 확립해 둠. 북마크도 그대로 따라 기본값 없이 깔끔한 URL.
3. **프리셋의 즉각적 가치** — jupiter-x10 + belt=300 조합으로 내행성 섭동이 몇 초만에 체감되는 교육 효과.
4. **브라우저 3단계 자동화 안정** — verify-p2c 1명령으로 UI 3기능 전부 회귀 감지.

## 어려웠던 것

1. **서버 포트 충돌** — Playwright 검증 중 `next start`가 EADDRINUSE로 실패. 사용한 PID 추적 대신 `pkill -f "next start"` 패턴 정립.
2. **jsdom `replaceState` URL origin** — bookmark-button 테스트에서 `http://localhost/...` 쓰기가 SecurityError. 상대 경로 `/ko`로 전환.
3. **MassSlider 위치 결정** — 처음엔 CelestialInfoPanel 안에 넣었으나 천체 미선택 시 프리셋까지 사라지는 문제. ScenarioPresets는 SidePanels 레벨로 분리.

## 프로세스 교훈

- **차분 구독 패턴** — store 구독 callback에서 prev와 비교하여 제거된 키도 복원 전파 (#107 sim-canvas). Zustand `subscribe((next, prev))` 시그니처 활용 안정적.
- **"리셋" 버튼은 기본값 교체가 명확한 UX** — 슬라이더 0.1~10 범위에서 "1.0으로 되돌리기"는 시각적으로 애매. 별도 리셋 버튼이 더 직관적.
- **프리셋 + 기존 기능 연계** — `?belt=300`과 jupiter-x10 조합처럼 기능 단위로는 드라이하던 것들이 조합에서 살아남. 추후 교육 콘텐츠 작성 시 참고.

## P2-D 인수인계

### 즉시 가능

- JPL Horizons 1000년 궤도 검증 자동화 (Newton N-body)
- 실 브라우저 환경 성능 측정 (`?belt=1000` 60fps 가능성 확인)
- v0.2.0 태그 + CHANGELOG

### 기술 부채

- 소행성대 + Newton 엔진 통합 불가 (현재 belt는 Kepler 전용) — P3 GPU compute에서 재검토
- 프리셋 시간 리셋 로직이 setTimeout 없이 직접 sendCommand — 전환 중 race condition 가능성 낮지만 관찰 필요
- 북마크 버튼의 폴백 `window.prompt`는 UX가 거칠음 — Clipboard API 미지원은 희귀하지만 대체 표시 고려

### 설계 검토 필요

- 질량 변경 시점부터 시간 역행으로 원 상태 복원 불가 → "프리셋 원복"이 암묵적 리셋 역할. 명시적 "시간 역행 불가" 힌트 필요 여부.
- 프리셋 확장 시 드롭다운 vs 버튼 그룹 — 현재 3개는 버튼으로 OK, 10개 넘어가면 UI 재설계.

## 결론

P2-C는 P2-A의 Newton 엔진에 사용자 통제권을 부여한 단계. **질량 슬라이더**(#107) + **URL 북마크**(#108) + **프리셋**(#109)의 3축으로 "만약에" 시나리오 탐구 루프를 완성했다. 성능 영향은 무시할 수준이며, P2-D에서 실 브라우저 측정과 v0.2.0 릴리스 준비로 넘어간다.
