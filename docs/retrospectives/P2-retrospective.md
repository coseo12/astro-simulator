## 기술 성과

### 잘 된 것

1. **Rust + wasm-pack 도입** — 1.94.1 pin + 조건부 exports(node/bundler). 첫 설치부터 CI까지 2시간 내 안착.
2. **Velocity-Verlet 정확도** — 목표(1년 <0.1%) 대비 실측 6자리수 여유(1e-5 ~ 1e-6).
3. **심플렉틱 역행 대칭성** — <1e-9 상대 오차. 고정 dt 균등 분할이 bit-exact 복원 보장.
4. **ThinInstances의 결정적 이득** — 실 GPU에서 N=1000 소행성대가 vsync cap 도달. P1 구조의 단일 draw call 선택이 이후 확장에 핵심적.
5. **차분 구독 패턴** — Zustand `subscribe((next, prev))`로 store 변경을 씬에 안정적으로 전파. `massMultipliers` 삭제 키도 복원.
6. **프로세스 가드의 효과** — P2-0의 PR 템플릿 + verify:test-coverage로 P1에서 반복된 UI 검증 누락·Vitest 미설치 사고가 P2에서 0건.

### 어려웠던 것

1. **첫 번째 #87 시도 오류** — 전체 N-body vs 2-body Kepler 비교로 Mercury 3% 오차 발견. 알고리즘 오류가 아닌 물리적 섭동 차이를 나중에 인지하고, 2-body Newton만 분리해 재설계.
2. **Mercury 이심률 + 짧은 주기** — dt=1d에서 위상 오차 수 % 누적. dt=10min으로 조정했지만 일부 물리 검증은 임계값 완화 필요(5e-3).
3. **헤드리스 측정 한계** — swiftshader 소프트웨어 렌더로는 실 성능 평가 불가. ANGLE Metal 플래그 조합 발견까지 시행착오.
4. **서버 포트 충돌** — `next start` 프로세스가 종료 신호 이후에도 3001 점유. `pkill -9 -f "next-server"` 패턴 필수.
5. **axe 위반 발견 지연** — MassSlider aria-label / canvas tabindex=1 같은 접근성 이슈가 P2-D 재검증에서 처음 드러남. 각 UI PR에 a11y 체크를 통합할 필요.

## 프로세스 교훈

- **"전면 Newton"의 의미 확장** — 단순히 엔진 교체가 아니라, 엔진 간 심리스 전환·질량 파라미터·URL 공유까지 한 루프로 묶임. 사용자 탐구가 실제 학습으로 이어짐.
- **바디 수 증가 = 컨텐츠 아니라 비용** — 8 → 18 바디로 fps 20% 감소. orbit 샘플링·per-frame 스케일 계산이 선형으로 누적. "한 번 한 계산"을 찾아 WASM으로 옮기는 노력이 P3 필수.
- **브라우저 3단계 검증 자동화의 재사용** — 각 마일스톤마다 `verify-<feature>.mjs` 작성, `verify-p2X.mjs`로 일괄 실행. 6개월 뒤 회귀 감지에 즉시 사용.
- **교차 마일스톤 의존성 관리** — P2-B 소행성대가 P2-C에서 "목성 10× + belt=300" 교육 시나리오의 기반이 됨. 단독 기능이 조합에서 가치 발현.
- **프리셋의 교육적 가치** — jupiter-x10 + belt=300 조합으로 내행성 섭동이 수 초 만에 체감. 수동 파라미터 조절보다 학습 문턱 낮음.

## P3 인수인계

### 즉시 가능

- WebGPU compute shader로 Kepler positionAt batch — 소행성대 N=10000 목표
- Barnes-Hut octree (N-body O(N log N)) 구현
- 모바일(iOS Safari / Android Chrome) 실 성능 측정
- 카이퍼대 / 오르트 구름 데이터 확장

### 기술 부채

- 소행성대는 Kepler 전용 — 대형 body(질량 변경)로 belt 섭동 시뮬은 불가
- 혜성 비중력 효과(태양풍) 없음 — Halley 예측 정확도 한계
- per-body 스케일 전환이 부드럽지 않음 — smooth easing 검토

### 설계 검토 필요

- 대규모 N-body에서 부동소수점 RMS 드리프트 누적 — fixed-point / Kahan summation 고려
- GPU compute reduction 결정성 — 시간 역행 대칭성 유지 위해 순서 고정 필요(#88 문서 참고)
- 파라미터 변경 후 "되돌리기" UX — 프리셋 원복이 암묵 리셋. 명시적 히스토리 UI 필요?

## 결론

P2는 **Kepler 해석해 기반 시뮬레이터를 Newton N-body + 사용자 통제가 가능한 탐구 도구로 전환**한 단계다. 24개 이슈를 5개 마일스톤에 순차 배치했고, 각 단계마다 검증 자동화와 회고를 병행해 P1 회고에서 식별된 프로세스 사고(UI 검증 누락, Vitest 누락) 재발 없이 완료했다.

성능은 실 GPU 환경에서 목표를 크게 상회했고(N=1000 120fps), 대신 헤드리스 측정이 현실의 프록시로 부적합함을 확인했다. P3에서는 WebGPU compute + 모바일을 실 성능 기준으로 재정의한다.
