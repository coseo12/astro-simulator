## 기술 성과

### 잘 된 것

1. **데이터 정확성** — Standish/Meeus 레퍼런스 대비 ±1% 검증 자동화
2. **이중 레이어 설계** — ESLint로 core에 React import 금지 강제. 리팩토링 안전성 확보
3. **RTE 좌표계** — 10^13m 스케일에서 1m 정밀도 단위 테스트로 검증
4. **Tier 배지** — 모든 수치에 데이터 신뢰성 표시 (교육적 정직성)
5. **모드별 토큰 스왑** — `data-mode` + CSS variables로 재렌더 없이 전환
6. **브라우저 3단계 검증 자동화** — Playwright 스크립트 5종 정립

### 어려웠던 것

1. **CRITICAL #3 위반 인지 지연** — Group B 전후로 각 UI PR마다 브라우저 검증을 했어야 했으나 "E2로 미룬다"고 변명했다가 사용자 지적으로 교정
2. **테스트 없이 PR 머지** — apps/web Vitest 설정을 빠뜨려 UI 단위 테스트 0건 상태로 D 그룹 전부 머지 후 사용자 지적으로 교정 (fix PR #68)
3. **WebGPU headless 한계** — Playwright Chromium은 WebGPU 미지원, adapter 사전 체크로 로그 오염 해결
4. **URL sync 무한 루프** — julianDate 매 프레임 URL 쓰기 → React 렌더 폭주. 북마크 방식으로 전환
5. **행성 시각 크기** — 실제 크기로는 점. × 500 배율 사용 시 Moon이 Earth 내부에 위치 (per-body 스케일 P2)

## 프로세스 교훈

- **CRITICAL DIRECTIVES 재인지** — 세션 진행 중 규칙을 자동으로 의식하지 못함. 각 UI PR 생성 전 자기 점검 필수
- **테스트 인프라 우선** — 프로젝트 스캐폴딩(A6) 단계에서 모든 패키지에 테스트 설정을 넣어야 했음. apps/web 누락이 후속 9개 PR에서 한 번도 검증 안 된 원인
- **배치 PR의 함정** — 여러 이슈를 한 PR에 묶을수록 회고 품질 저하. Group D에서 2~3개씩 묶은 것은 효율적이었으나 개별 DoD 추적 어려움
- **검증 스크립트 일관성** — 초기에는 매번 수정했으나 `verify:all` 체계화 후 안정적

## P2 인수인계

### 즉시 가능

- 심플렉틱 적분기 (Leapfrog/Verlet) 구현 → N-body 전환
- 소행성대 대표 샘플 + 혜성 (핼리 등) 추가
- 거리-의존 per-body 시각 스케일 (Moon 가시화)
- 시나리오 URL 북마크 (시간 포함)

### 기술 부채

- core 내 solar-system-scene의 `updateAt`이 매 프레임 Map 재생성 — 캐시 필요
- orbit 라인을 InstancedMesh 또는 LineSystem으로 최적화 (현재 Mesh per orbit)
- WebGPU 경로 실환경 CI (GPU 가속 가능한 헤드리스 환경 탐색)
- 로그 타임라인 스크러버 (지연)

### 설계 검토 필요

- Floating Origin 통합 시점 — 현재는 AU 씬 단위라 불필요. 표면 수준 줌 요구 시 재검토
- 시간 역행 중 심플렉틱 적분기 대칭성 검증
- 단위 토글 실제 포매팅 연동 (InfoPanel 숫자가 단위 설정에 반응)

## 결론

P1 MVP는 **스프린트 계약 기준 모든 핵심 항목을 달성**했다.
브라우저에서 실제로 **태양계 전체가 정확하게 움직이고, 포커스 전환이 자연스러우며, 89개 단위 테스트로 궤도 정확성이 보장**된다.

규칙 위반 2건(UI 테스트 / 브라우저 검증)은 사용자 지적으로 세션 내 교정됐으며, 이후 전 Phase에 적용할 프로세스 가이드를 문서화했다.
