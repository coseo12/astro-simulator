## 1. 프로젝트 개요

### 1.1 비전
브라우저에서 동작하는 **정밀한 천체물리 시뮬레이터**. 태양계를 기점으로 관측가능우주까지 연속 스케일로 탐험 가능하며, 관측 데이터 기반의 정확성과 가상 실험의 자유도를 동시에 제공한다.

### 1.2 지향점 (4개 동시 추구)
1. **교육 도구** — 천체물리 개념을 정확한 시각화로 전달
2. **연구 시각화** — 실제 카탈로그 기반 데이터 탐색
3. **몰입형 탐험** — 스케일 연속성, 시각적 완성도
4. **물리 샌드박스** — 사용자 실험, "만약에" 시나리오

### 1.3 핵심 차별점
- 대부분의 웹 천체 시뮬레이터는 단일 스케일(태양계 OR 은하) 또는 정적 시각화에 머무름
- 본 프로젝트는 **연속 스케일 + 물리 정확도 + 확장 가능한 샌드박스**를 결합
- 프리컴퓨트 영상이 아닌 **실시간 연산** 기반

## 3. 핵심 기능 (Feature Set)

### 3.1 탐색·관찰
- 태양계 → 은하 → 우주 연속 줌
- 시간 가속/감속/역행 (1초 = 1일~1억년 범위)
- 특정 날짜 배치 재현 (과거/미래)
- 천체 선택 시 기준 카메라 전환
- 천체 정보 패널 (데이터 신뢰성 티어 표시)

### 3.2 물리 시뮬레이션
- Kepler 해석궤도 (P1)
- Newton N-body + 심플렉틱 적분기 (P2+)
- WebGPU Compute 대규모 N-body (P3+)
- 일반상대론 효과 (P4+) — 수성 근일점, 중력렌즈
- 항성 진화 상태머신 (P5+)

### 3.3 데이터 연동
- JPL Horizons (태양계)
- Gaia DR3 (항성 20억)
- NASA Exoplanet Archive (외계행성)
- SDSS (은하)
- Planck (CMB)

### 3.4 인터랙션 모드
- **관찰 모드**: 기본, 수동적 탐색
- **교육 모드**: 설명 오버레이, 가이드 투어
- **연구 모드**: 정밀 수치, CSV 내보내기, 측정 도구
- **샌드박스 모드**: 사용자 천체 생성/파괴, 시나리오 저장/공유

### 3.5 확장 기능
- WebXR (VR 탐험)
- 다국어 (한국어 우선, 영어)
- 반응형 (데스크톱 우선, 모바일은 경량 모드)

## 4. 기술 아키텍처

상세: `docs/phases/architecture.md` 참조.

### 4.1 스택 요약
| 영역 | 선택 |
|---|---|
| 렌더 엔진 | Babylon.js (WebGPU-first, WebGL2 폴백) |
| UI 프레임워크 | Next.js (App Router) |
| 언어 | TypeScript |
| 고성능 연산 | Rust → WebAssembly (P2+) |
| 패키지 매니저 | pnpm (모노레포) |
| 테스트 | Vitest + Playwright |
| 배포 | Vercel (초기), 추후 Cloudflare Pages 검토 |

### 4.2 프로젝트 구조
```
/apps/web              Next.js 애플리케이션 (UI 레이어)
/packages/core         @space/core — 순수 TS 시뮬레이션 코어
/packages/shared       공용 타입/상수/이벤트 정의
/packages/data         천체 데이터 로더/캐시
/docs                  기획/아키텍처/Phase 문서
```

### 4.3 이중 레이어 원칙
- **코어**: React/Next 의존성 제로. 어떤 UI 프레임워크와도 연동 가능
- **UI**: 읽기 전용 구독자 + 명령 발행자. 시뮬레이션 상태를 소유하지 않음
- **통신**: EventEmitter 기반 명령/이벤트

### 4.4 핵심 기술 결정
- **좌표계**: CPU float64 월드 + GPU RTE(Relative-to-Eye) float32
- **적분기**: Leapfrog/Verlet 심플렉틱 (에너지 보존)
- **GPU 전략**: GPU-resident state, readback 최소화
- **렌더 기법**: Logarithmic depth buffer, 저해상도 compute + TAA

| Phase | 범위 | 완결 제품 |

## 6. 개발 방식

### 6.1 작업 원칙
- **스프린트 계약**: 각 Phase 착수 전 검증 가능한 완료 기준 합의
- **브라우저 3단계 검증**: 정적 → 인터랙션 → 흐름. 빌드 통과만으로 "완료" 판정 금지
- **AI 페어 프로그래밍**: Claude 주도 구현, Gemini 교차검증
- **점진적 확장**: 진행하며 수정. 사전 과잉 설계 금지

### 6.2 브랜치 전략
- `main`: 안정 배포
- `develop`: 통합
- `feature/<이슈번호>-<설명>`: 기능 개발
- `fix/<이슈번호>-<설명>`: 버그 수정
- PR 필수, main 직접 푸시 금지

### 6.3 커밋 컨벤션
```
<type>(<scope>): <description>
type: feat | fix | refactor | test | docs | chore
```

### 6.4 릴리스
- Semantic Versioning
- Phase 완료 시 `git tag` + GitHub Release

### 6.5 문서화 의무
- Phase 착수 시: 스프린트 계약서 작성 (`docs/phases/P<N>-*.md`)
- Phase 완료 시: 회고 작성 (`docs/retrospectives/`)
- 아키텍처 변경 시: ADR 업데이트

## 8. 리스크 관리

### 8.1 기술 리스크
| 리스크 | 영향 | 대응 |
|---|---|---|
| WebGPU 브라우저 파편화 | 중 | WebGL2 폴백 유지, 주요 브라우저 주기적 테스트 |
| float 정밀도 한계 | 높음 | RTE + Floating Origin으로 해결, P1에서 검증 |
| 대규모 N-body 성능 | 높음 | WebGPU Compute + 공간 해싱, 최악 시 LOD |
| 브라우저 메모리 한계 (2-4GB) | 중 | 스트리밍 로더, 프러스텀 컬링, LOD |
| 심플렉틱 적분기 구현 난이도 | 중 | 기존 논문/레퍼런스 코드 참조, 단위 테스트 |

### 8.2 프로젝트 리스크
| 리스크 | 영향 | 대응 |
|---|---|---|
| 1인 개발 지속성 | 높음 | Phase 단위 완결 제품 → 중단 시점 무관하게 결과물 존재 |
| 스코프 크리프 | 중 | Phase별 스코프 엄격 준수, 확장은 다음 Phase로 연기 |
| 데이터 소스 API 변경 | 낮음 | 주요 데이터 스냅샷 번들링 |
| 커뮤니티 레퍼런스 부족 (Babylon) | 낮음 | 공식 포럼 적극 활용, 필요 시 논문 직접 포팅 |

### 8.3 데이터 리스크
- 외부 API (JPL, Gaia 등) 의존성 → 캐싱/스냅샷 전략
- 라이선스: 모든 데이터 소스의 이용 약관 확인 후 채택
- 정확도 오표시 방지: 데이터 신뢰성 티어를 UI에 명시 (교육적 정직성)
